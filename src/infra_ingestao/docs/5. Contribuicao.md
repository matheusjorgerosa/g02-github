# 5. Guia de Contribuição — Infraestrutura já provisionada

Este guia é para quem vai **continuar o desenvolvimento** do projeto com a infraestrutura já rodando no GCP. Você **não precisa** executar `terraform apply` nem recriar nada do zero — tudo já está de pé.

---

## Visão geral da arquitetura

```
POST /ingest
    → API Gateway (us-east1)
        → Cloud Run "producer" (Go) → Pub/Sub "ingestao-topic"

Cloud Scheduler (a cada 5 min)
    → Cloud Run Job "consumer" (Go)
        → lê Pub/Sub, comprime em .json.gz
        → escreve em gs://...-venus-datalake/raw/YYYY/MM/DD/batch-{ts}.json.gz

Eventarc (object.finalized no bucket)
    → Cloud Run "transformer" (Python/Flask)
        → lê raw/.json.gz
        → escreve Parquet em gs://...-venus-datalake/trusted/ (Apache Iceberg)
        → atualiza catálogo no Cloud SQL (PostgreSQL)

BigQuery (tabela externa)
    → lê Parquet de trusted/default.db/ingestao/data/*
```

**Projeto:** `venus-m09`  
**Região:** `southamerica-east1`  
**Bucket:** `southamerica-east1-venus-datalake`  
**Artifact Registry:** `southamerica-east1-docker.pkg.dev/venus-m09/venus-images`

## 1. Alterando código dos serviços

Quando você modifica o código de qualquer serviço (producer, consumer ou transformer), precisa **reconstruir a imagem Docker** e **fazer deploy** para que a mudança tenha efeito.

### Producer (Go)

```bash
# Na raiz do projeto (infra_ingestao/)
cd producer

# Build e push
docker build --platform linux/amd64 -t southamerica-east1-docker.pkg.dev/venus-m09/venus-images/producer:latest .
docker push southamerica-east1-docker.pkg.dev/venus-m09/venus-images/producer:latest

# Forçar o Cloud Run a puxar a nova imagem
gcloud run services update producer \
  --image southamerica-east1-docker.pkg.dev/venus-m09/venus-images/producer:latest \
  --region southamerica-east1 \
  --project venus-m09
```

### Consumer (Go)

```bash
cd consumer

docker build --platform linux/amd64 -t southamerica-east1-docker.pkg.dev/venus-m09/venus-images/consumer:latest .
docker push southamerica-east1-docker.pkg.dev/venus-m09/venus-images/consumer:latest

# Para Jobs, use "update" no job:
gcloud run jobs update consumer \
  --image southamerica-east1-docker.pkg.dev/venus-m09/venus-images/consumer:latest \
  --region southamerica-east1 \
  --project venus-m09
```

### Transformer (Python)

```bash
cd transformer

docker build --platform linux/amd64 -t southamerica-east1-docker.pkg.dev/venus-m09/venus-images/transformer:latest .
docker push southamerica-east1-docker.pkg.dev/venus-m09/venus-images/transformer:latest

gcloud run services update transformer \
  --image southamerica-east1-docker.pkg.dev/venus-m09/venus-images/transformer:latest \
  --region southamerica-east1 \
  --project venus-m09
```

> **Dica:** Antes de fazer `docker push`, autentique no Artifact Registry:
> ```bash
> gcloud auth configure-docker southamerica-east1-docker.pkg.dev
> ```

## 2. Alterando variáveis de ambiente

As variáveis de ambiente dos serviços são definidas no `main.tf`, mas você pode alterá-las diretamente pelo CLI sem precisar rodar Terraform.

### Exemplo: mudar o BATCH_SIZE do consumer (padrão: 100)

```bash
gcloud run jobs update consumer \
  --set-env-vars BATCH_SIZE=50 \
  --region southamerica-east1 \
  --project venus-m09
```

### Exemplo: mudar variáveis do transformer

```bash
gcloud run services update transformer \
  --set-env-vars DB_USER=iceberg,DB_PASS=nova_senha,DB_NAME=iceberg \
  --region southamerica-east1 \
  --project venus-m09
```

> **Importante:** Se alterar via CLI, o Terraform ficará dessincronizado. Na próxima vez que alguém rodar `terraform apply`, as variáveis voltarão aos valores do `.tf`. Prefira alterar no `main.tf` quando for uma mudança definitiva.

### Variáveis de ambiente por serviço

| Serviço | Variável | Descrição |
|---|---|---|
| **producer** | `PUBSUB_TOPIC` | Nome do tópico Pub/Sub (`ingestao-topic`) |
| **producer** | `GCP_PROJECT` | ID do projeto (`venus-m09`) |
| **consumer** | `GCP_PROJECT` | ID do projeto |
| **consumer** | `PUBSUB_SUBSCRIPTION` | Nome da subscription (`ingestao-sub`) |
| **consumer** | `GCS_BUCKET` | Nome do bucket do Data Lake |
| **consumer** | `BATCH_SIZE` | Quantidade de mensagens por batch (padrão: `100`) |
| **transformer** | `GCP_PROJECT` | ID do projeto |
| **transformer** | `GCS_BUCKET` | Nome do bucket |
| **transformer** | `ICEBERG_WAREHOUSE` | Caminho GCS do warehouse Iceberg (`gs://.../trusted`) |
| **transformer** | `DB_USER` | Usuário do Cloud SQL (`iceberg`) |
| **transformer** | `DB_PASS` | Senha do Cloud SQL |
| **transformer** | `DB_NAME` | Nome do banco (`iceberg`) |
| **transformer** | `INSTANCE_CONNECTION_NAME` | Connection name do Cloud SQL |

## 3. Consultando dados com BigQuery

Uma tabela externa no BigQuery já está configurada e aponta para os Parquet da camada trusted. Ela **se atualiza automaticamente** — cada novo arquivo Parquet escrito pelo transformer aparece nas queries sem ação manual.

### No Console GCP

1. Acesse [BigQuery Console](https://console.cloud.google.com/bigquery?project=venus-m09)
2. No painel esquerdo, expanda `venus-m09 → trusted → ingestao`
3. Clique em **Query** ou escreva SQL:

```sql
-- Total de registros
SELECT COUNT(*) FROM `venus-m09.trusted.ingestao`;

-- Últimos 10 registros
SELECT * FROM `venus-m09.trusted.ingestao` ORDER BY ingested_at DESC LIMIT 10;

-- Contagem por dia (partição Hive)
SELECT ingested_at__day, COUNT(*) as total
FROM `venus-m09.trusted.ingestao`
GROUP BY ingested_at__day
ORDER BY ingested_at__day DESC;
```

### Via CLI (bq)

```bash
bq query --use_legacy_sql=false \
  'SELECT COUNT(*) as total FROM `venus-m09.trusted.ingestao`'
```

## 4. Limpando o Data Lake (reset completo)

Se precisar apagar todos os dados e recomeçar do zero, siga **todos os 3 passos** na ordem. Apagar só os arquivos sem limpar o catálogo causa erros de `duplicate key` e `FileNotFoundError`.

### Passo 1 — Apagar arquivos do GCS

```bash
# Apagar camada raw
gsutil -m rm -r gs://southamerica-east1-venus-datalake/raw/**

# Apagar camada trusted (Parquet + metadata Iceberg)
gsutil -m rm -r gs://southamerica-east1-venus-datalake/trusted/**
```

### Passo 2 — Limpar o catálogo Iceberg (Cloud SQL)

Acesse o [Cloud SQL Studio](https://console.cloud.google.com/sql/instances/iceberg-catalog/studio?project=venus-m09):

1. Selecione o banco `iceberg`, usuário `iceberg`
2. Execute os comandos SQL:

```sql
-- Ver o que existe no catálogo
SELECT * FROM iceberg_tables;
SELECT * FROM iceberg_namespace_properties;

-- Apagar registros do catálogo
DELETE FROM iceberg_tables;
DELETE FROM iceberg_namespace_properties;
```

### Passo 3 — Recriar as pastas vazias no bucket

```bash
# O Terraform cria essas pastas, mas se você apagou com rm -r, recrie:
echo -n " " | gsutil cp - gs://southamerica-east1-venus-datalake/raw/
echo -n " " | gsutil cp - gs://southamerica-east1-venus-datalake/trusted/
```

> **Por que os 3 passos?** O catálogo Iceberg (PostgreSQL) mantém ponteiros para arquivos de metadata no GCS. Se você apaga os arquivos sem limpar o catálogo, o transformer tenta usar metadata que não existe mais → erro. Se limpa o catálogo sem apagar os arquivos, dados duplicados aparecem.

## 6. Modificando a infraestrutura (Terraform)

Se precisar alterar recursos de infraestrutura (ex: adicionar novo serviço, mudar configuração do Cloud SQL, etc.), edite o `main.tf` e aplique:

```bash
# Ver o que vai mudar ANTES de aplicar
terraform plan

# Aplicar as mudanças
terraform apply
```

### Arquivos importantes do Terraform

| Arquivo | Conteúdo |
|---|---|
| `main.tf` | Todos os recursos GCP (bucket, Pub/Sub, Cloud Run, Eventarc, Cloud SQL, BigQuery) |
| `variables.tf` | Definição das variáveis (`project_id`, `region`, `iceberg_db_password`) |
| `terraform.tfvars` | Valores das variáveis (contém senha — **não commitar**) |
| `providers.tf` | Providers Google e Google Beta |
| `outputs.tf` | Outputs com URLs e nomes dos recursos |
| `terraform.tfstate` | Estado atual da infra (**não editar manualmente, não commitar**) |

### Cuidados

- **Nunca** delete o `terraform.tfstate` — é o mapeamento entre o `.tf` e o que existe no GCP.
- Rode `terraform plan` antes de `apply` para revisar mudanças.
- Se alterar variáveis de ambiente via CLI (seção 2), lembre de atualizar o `main.tf` também.
- O `terraform.tfvars` contém a senha do banco — mantenha fora do controle de versão.

## 7. Troubleshooting rápido

| Sintoma | Causa provável | Solução |
|---|---|---|
| Transformer dá `duplicate key` | Catálogo PostgreSQL dessincronizado com GCS | Limpar catálogo (seção 4, passo 2) |
| Transformer dá `FileNotFoundError` | Metadata Iceberg apagado do GCS | Reset completo (seção 4, todos os passos) |
| Transformer dá `404 NotFound` | Eventarc disparou para arquivo da `trusted/` | Ignorar — o código já trata (retorna 200) |
| Consumer não roda | Scheduler pausado ou SA sem permissão | Verificar `gcloud scheduler jobs list` e IAM |
| BigQuery retorna 0 linhas | Nenhum Parquet escrito ainda na trusted | Enviar dados e aguardar o pipeline processar |
| `docker push` dá 403 | Não autenticou no Artifact Registry | `gcloud auth configure-docker southamerica-east1-docker.pkg.dev` |
| `terraform apply` dá conflito | State dessincronizado | `terraform refresh` e depois `terraform apply` |
