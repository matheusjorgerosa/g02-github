# Consumer Batch — Pub/Sub → GCS (raw)

Este documento descreve como subir o consumer que lê mensagens do Pub/Sub em batch, comprime com gzip e grava na camada `raw/` do Data Lake particionado por data.

---

## Arquitetura

```
Pub/Sub Subscription (ingestao-sub)
         ↓
Cloud Scheduler (a cada 5 min)
         ↓
Cloud Run Job (consumer - Go)
         ↓
GCS: raw/YYYY/MM/DD/batch-{unix_ts}.json.gz
```

---

## O que será criado

| Recurso | Nome |
|---|---|
| Service Account | `cloud-run-consumer` (com `pubsub.subscriber` + `storage.objectCreator` + `run.invoker`) |
| Cloud Run Job | `consumer` |
| Cloud Scheduler | `trigger-consumer-batch` (cron `*/5 * * * *`) |

---

## Pré-requisitos

- Infraestrutura anterior já rodando (bucket, Pub/Sub, producer)
- APIs habilitadas:

```bash
gcloud services enable \
  cloudscheduler.googleapis.com \
  run.googleapis.com
```

---

## Passo a passo

### 1. Resolver dependências do Go

```bash
cd consumer/
go mod tidy
cd ..
```

---

### 2. Buildar e fazer push da imagem Docker

```bash
cd consumer/

docker build -t southamerica-east1-docker.pkg.dev/venus-m09/venus-images/consumer:latest .

docker push southamerica-east1-docker.pkg.dev/venus-m09/venus-images/consumer:latest

cd ..
```

---

### 3. Subir a infra do consumer

```bash
terraform apply \
  -target=google_service_account.cloud_run_consumer \
  -target=google_project_iam_member.consumer_pubsub_subscriber \
  -target=google_project_iam_member.consumer_storage_writer \
  -target=google_project_iam_member.consumer_run_invoker \
  -target=google_cloud_run_v2_job.consumer \
  -target=google_cloud_scheduler_job.trigger_consumer
```

Ou, se toda a infra anterior já existe:

```bash
terraform apply
```

---

### 4. Verificar os outputs

```bash
terraform output consumer_job_name
terraform output scheduler_job_name
```

## Como funciona

### Fluxo do consumer (`consumer/main.go`)

1. Conecta no Pub/Sub e no GCS via SDKs
2. Faz pull síncrono de até `BATCH_SIZE` mensagens (padrão: 100) com timeout de 30s
3. Se não houver mensagens, encerra sem escrever nada
4. Serializa todas as mensagens como um JSON array
5. Comprime com gzip
6. Grava em `raw/YYYY/MM/DD/batch-{unix_timestamp}.json.gz`
7. Encerra (exit 0)

### Variáveis de ambiente (injetadas pelo Terraform)

| Variável | Valor |
|---|---|
| `GCP_PROJECT` | `venus-m09` |
| `PUBSUB_SUBSCRIPTION` | `ingestao-sub` |
| `GCS_BUCKET` | `southamerica-east1-venus-datalake` |
| `BATCH_SIZE` | `100` |

### Cloud Scheduler

- Cron: `*/5 * * * *` (a cada 5 minutos)
- Timezone: `America/Sao_Paulo`
- Autentica via OAuth com a Service Account `cloud-run-consumer`
- Invoca a API do Cloud Run para executar o job

## Particionamento

Os arquivos ficam organizados por data de execução:

```
raw/
├── 2026/
│   ├── 03/
│   │   ├── 03/
│   │   │   ├── batch-1741024500.json.gz
│   │   │   └── batch-1741024800.json.gz
│   │   ├── 04/
│   │   │   └── batch-1741110900.json.gz
```

Cada arquivo contém um JSON array com até `BATCH_SIZE` mensagens, comprimido com gzip.

---

## Notas

- Se não houver mensagens na fila, o job roda e encerra sem escrever nada no bucket
- O timeout do job é 120s — suficiente para pull + compress + upload
- O Scheduler só dispara se o App Engine está habilitado na região (requisito do Cloud Scheduler). Se der erro, habilite: `gcloud app create --region=southamerica-east1`
- `BATCH_SIZE` é configurável via variável de ambiente no `main.tf`
