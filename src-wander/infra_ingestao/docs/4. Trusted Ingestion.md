# Transformer — Raw → Trusted (Eventarc + Cloud Run + Iceberg/Parquet)

Este documento descreve como subir o **transformer**, responsável por ler arquivos JSON comprimidos da camada `raw/`, realizar processamento básico (deduplicação), e gravar como **Parquet** em uma **tabela Iceberg** na camada `trusted/` do Data Lake.

---

## Arquitetura

```
GCS: raw/YYYY/MM/DD/batch-*.json.gz
         ↓ (google.cloud.storage.object.v1.finalized)
Eventarc Trigger (raw-file-created)
         ↓
Cloud Run Service (transformer - Python)
    ├── Lê JSON comprimido da raw/
    ├── Deduplica registros
    ├── Converte para PyArrow Table
    ├── Append na tabela Iceberg (Parquet)
    └── Load no BigQuery (tabela nativa, dual-write)
         ↓
         ├── GCS: trusted/ (tabela Iceberg com arquivos Parquet + metadata)
         ├── Cloud SQL PostgreSQL (catálogo Iceberg)
         └── BigQuery: trusted.ingestao (tabela nativa, particionada por ingested_at)
```

**Cada arquivo criado na `raw/` gera uma invocação independente do Cloud Run.** Em alto fluxo, múltiplas instâncias rodam em paralelo (até 5), cada uma processando 1 arquivo. Isso é possível porque o catálogo Iceberg roda em PostgreSQL (Cloud SQL), que suporta commits concorrentes.

## Por que Python (e não Go)?

O consumer e o producer estão em Go — e faz sentido, pois são I/O simples (HTTP → Pub/Sub, Pub/Sub → GCS). Já o transformer envolve **processamento de dados**:

- **Apache Iceberg**: a biblioteca oficial é `pyiceberg` (Python). Go **não tem** lib oficial.
- **Parquet**: `pyarrow` é a referência para leitura/escrita de Parquet, com performance C++ por baixo.
- **Ecossistema**: pandas, polars, pyarrow — todo o tooling de data engineering é Python-first.

---

## Por que Eventarc (e não Cloud Scheduler)?

O consumer da `raw/` roda a cada 5 min via Cloud Scheduler. Para a trusted, usamos **Eventarc** (event-driven):

| Aspecto | Cloud Scheduler (time-based) | Eventarc (event-driven) |
|---|---|---|
| Trigger | A cada N minutos | Quando arquivo é criado |
| Runs vazios | Sim (se não houver arquivo novo) | Não |
| Latência | Até N minutos | Segundos após criação |
| State management | Precisa rastrear quais arquivos já processou | Não — 1 evento = 1 arquivo |
| Complexidade | Precisa listar + filtrar objetos | Recebe o nome do arquivo no evento |

---

## Por que Cloud SQL para o catálogo Iceberg?

O catálogo Iceberg gerencia **metadados** da tabela: schemas, snapshots, partitions, e ponteiros para os arquivos Parquet. Opções consideradas:

| Opção | Concorrência | Persistência | Custo |
|---|---|---|---|
| SQLite em `/tmp` | ❌ Efêmero | ❌ Perde entre instâncias | Zero |
| SQLite no GCS | ❌ Corrompe | ✅ | Zero |
| **Cloud SQL PostgreSQL** | **✅ Suporta** | **✅** | **~$7-9/mês** |
| REST Catalog (Nessie) | ✅ | ✅ | Precisa manter outro serviço |

**Cloud SQL foi escolhido** porque o Eventarc pode disparar múltiplas instâncias simultâneas do Cloud Run, e o PostgreSQL suporta commits concorrentes sem problema. O `db-f1-micro` é suficiente para metadados Iceberg (tabelas pequenas) e custa ~$7/mês.

A conexão entre Cloud Run e Cloud SQL usa o **Cloud SQL Python Connector** (`cloud-sql-python-connector`), que não requer VPC Connector (economiza ~$7/mês adicionais).

## Schema da tabela Iceberg

```
Tabela: default.ingestao
Warehouse: gs://{bucket}/trusted/

Colunas:
  1. event_data   (string)     — JSON serializado do registro original
  2. ingested_at  (timestamp)  — momento do processamento (UTC)
  3. source_file  (string)     — caminho do arquivo raw de origem

Partição: ingested_day (DayTransform sobre ingested_at)
```

> **Nota:** O schema atual armazena o JSON inteiro como string. Quando o schema dos dados estabilizar, expanda as colunas (ex: `sensor`, `valor`, `timestamp`) para permitir queries colunares eficientes no Parquet.

## Como funciona (detalhado)

### Fluxo do transformer (`transformer/main.py`)

1. **Recebe CloudEvent** do Eventarc (tipo `google.cloud.storage.object.v1.finalized`)
2. **Extrai** `bucket` e `name` do payload do evento
3. **Filtra**: ignora objetos que não estejam em `raw/` ou não terminem em `.json.gz`
4. **Baixa** o arquivo do GCS via `google-cloud-storage`
5. **Descompacta** (gzip) e parseia o JSON array
6. **Deduplica** registros por conteúdo completo (serialização JSON canônica)
7. **Converte** para `pyarrow.Table` com colunas: `event_data`, `ingested_at`, `source_file`
8. **Conecta** ao catálogo Iceberg (PostgreSQL via Cloud SQL Connector)
9. **Cria** namespace e tabela se não existirem (idempotente)
10. **Faz append** dos dados na tabela Iceberg (gera arquivo Parquet + atualiza metadata)
11. **Faz load** dos mesmos dados no BigQuery (tabela nativa, dual-write). Se falhar, os dados já estão salvos no Iceberg
12. **Retorna** HTTP 200 com estatísticas (registros lidos, escritos, duplicatas removidas)

### Catálogo Iceberg (Cloud SQL)

- Tipo: `SqlCatalog` do PyIceberg
- Backend: PostgreSQL 15 (`db-f1-micro`) no Cloud SQL
- Conexão: via `cloud-sql-python-connector` (sem VPC Connector)
- Tabelas internas: o PyIceberg cria automaticamente tabelas de controle no banco (`iceberg_tables`, `iceberg_namespace_properties`)
- Warehouse: os dados (Parquet) e metadata (JSON) ficam no GCS em `gs://{bucket}/trusted/`

### Concorrência

- O Eventarc dispara uma requisição HTTP por arquivo criado na raw/
- O Cloud Run escala automaticamente até `max_instance_count=5`
- Cada instância processa 1 arquivo independentemente
- O PostgreSQL garante commits atômicos — sem conflito entre instâncias concorrentes
- O Iceberg usa **optimistic concurrency**: se duas instâncias tentarem commitar ao mesmo tempo, uma retenta automaticamente

## BigQuery — Consulta à camada trusted (tabela nativa, dual-write)

O BigQuery está configurado com uma **tabela nativa** particionada por `ingested_at`. O **transformer** faz dual-write: após o append no Iceberg, carrega os mesmos dados no BigQuery via `load_table_from_arrow`. Isso garante que os dados aparecem no BigQuery **segundos após o processamento**, sem dependência de metadata files externos.

> **Iceberg continua sendo o source of truth.** Se o load no BigQuery falhar, os dados estão seguros no Iceberg (GCS + Cloud SQL). O BigQuery é uma cópia otimizada para queries.

### Recursos Terraform

| Recurso | Nome / ID |
|---|---|
| IAM (SA) | `roles/bigquery.dataEditor` (escrever na tabela) |
| IAM (SA) | `roles/bigquery.jobUser` (executar jobs de load) |
| BigQuery Dataset | `trusted` (região `southamerica-east1`) |
| BigQuery Table | `ingestao` (nativa, particionada por `ingested_at` / DAY) |

### Como funciona

1. O **transformer** processa o arquivo raw e faz append no Iceberg
2. Logo em seguida, carrega os mesmos dados no BigQuery (`WRITE_APPEND`)
3. A coluna `ingested_day` (DATE) é adicionada automaticamente pelo transformer
4. O BigQuery particiona por `ingested_at` (DAY) para partition pruning
5. Se o load falhar, o erro é logado mas o request não falha (Iceberg já salvou)

### Consultar no BigQuery Console

Acesse: https://console.cloud.google.com/bigquery?project=venus-m09

```sql
-- Todos os registros
SELECT * FROM `venus-m09.trusted.ingestao` LIMIT 10;

-- Contagem total
SELECT COUNT(*) as total FROM `venus-m09.trusted.ingestao`;

-- Contagem por arquivo de origem
SELECT 
  source_file,
  COUNT(*) as registros,
  MIN(ingested_at) as primeiro,
  MAX(ingested_at) as ultimo
FROM `venus-m09.trusted.ingestao`
GROUP BY source_file
ORDER BY primeiro DESC;

-- Extrair campos do JSON (com filtro por partição para performance)
SELECT 
  JSON_VALUE(event_data, '$.sensor') as sensor,
  CAST(JSON_VALUE(event_data, '$.valor') AS FLOAT64) as valor,
  JSON_VALUE(event_data, '$.timestamp') as event_timestamp,
  ingested_at,
  ingested_day
FROM `venus-m09.trusted.ingestao`
WHERE ingested_day = CURRENT_DATE()
ORDER BY ingested_at DESC
LIMIT 20;
```

> **Dica de performance:** Use `WHERE ingested_at >= TIMESTAMP('2026-03-04')` ou `WHERE DATE(ingested_at) = '2026-03-04'` nas queries para aproveitar o partition pruning nativo (tabela particionada por `ingested_at` / DAY).

## Troubleshooting — Problemas conhecidos e soluções

### 1. `ImportError: cannot import name 'SQLALCHEMY_CATALOG_TABLE_BASE'`

**Causa:** O pyiceberg 0.9.x renomeou `SQLALCHEMY_CATALOG_TABLE_BASE` para `SqlCatalogBaseTable`. O código original usava a API interna da versão anterior.

**Solução:** O `main.py` atual usa `CloudSqlCatalog`, uma subclass do `SqlCatalog` que:
- Aceita um engine externo (Cloud SQL Connector) em vez de criar a partir de URI
- Chama `MetastoreCatalog.__init__()` para pular a criação de engine do `SqlCatalog`
- Importa `SqlCatalogBaseTable` (nome correto na 0.9.x)

### 2. `DatabaseError: relation "iceberg_tables" does not exist`

**Causa:** O pyiceberg 0.9.x no `_ensure_tables_exist()` só captura `OperationalError` e `ProgrammingError`, mas o driver pg8000 lança `DatabaseError` quando a tabela não existe.

**Solução:** O `CloudSqlCatalog` override `_ensure_tables_exist()` para capturar também `DatabaseError`:

```python
except (OperationalError, ProgrammingError, DatabaseError):
    self.create_tables()
```

### 3. `ValueError: Mismatch in fields (required vs optional)`

**Causa:** O schema Iceberg define `ingested_at` e `source_file` como `required`, mas o PyArrow criava colunas `nullable=True` por padrão.

**Solução:** O `transform_and_write()` define um `arrow_schema` explícito com `nullable=False` para os campos required:

```python
arrow_schema = pa.schema([
    pa.field("event_data", pa.string(), nullable=True),
    pa.field("ingested_at", pa.timestamp("us"), nullable=False),
    pa.field("source_file", pa.string(), nullable=False),
])
```

### 4. `404 Not Found` ao baixar arquivo da raw

**Causa:** O Eventarc disparou para um objeto que não existe mais (evento espúrio ou race condition no upload).

**Solução:** O handler captura `google.api_core.exceptions.NotFound` e retorna HTTP 200 para evitar retries infinitos do Eventarc:

```python
except NotFound:
    logger.warning(f"Arquivo não encontrado (404), ignorando: {object_name}")
    return jsonify({"status": "not_found", "object": object_name}), 200
```

### 5. `duplicate key value violates unique constraint "iceberg_tables_pkey"`

**Causa:** O catálogo PostgreSQL aponta para um metadata JSON que foi deletado do GCS. O `load_table()` falha com `FileNotFoundError`, o `except` genérico captura, e o `create_table()` falha porque a entrada já existe no banco.

**Solução:** O `get_or_create_table()` trata `FileNotFoundError` separadamente — dropa a tabela corrompida e recria. Veja a seção "Comandos Úteis" abaixo para limpeza manual.

### 6. Eventarc não suporta filtro por path/subject

O Eventarc **não permite** filtrar por `subject` (prefixo do objeto) em eventos `google.cloud.storage.object.v1.finalized`. O trigger dispara para **qualquer** objeto criado no bucket. O filtro é feito no código Python:

```python
if not object_name.startswith("raw/") or not object_name.endswith(".json.gz"):
    return jsonify({"status": "ignored"}), 200
```

O custo é mínimo — o handler retorna 200 imediatamente sem processamento.
